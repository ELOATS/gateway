# Service Contract

本文档定义 `logic-python` 当前稳定暴露给 Go 网关的服务契约。

## 1. 服务定位

`logic-python` 是增强能力服务，不承担主链路编排职责。它通过 gRPC 提供辅助能力，由 Go 端决定何时调用、如何降级、是否 fail-open 或 fail-closed。

## 2. 当前稳定 RPC

### CheckInput

用途：

- 对输入进行增强型检查。
- 可结合规则、分类器或轻量模型判断风险。

契约要求：

- 返回语义必须稳定。
- 不能隐式假设自己拥有最终阻断权。
- 如果行为语义发生变化，必须同步更新 Go 侧 facade 测试和文档。

### CheckOutput

用途：

- 对模型输出进行增强型检查或清洗。

契约要求：

- 同步与流式路径都应能获得一致的基本语义。
- 清洗或阻断行为必须对调用方可解释。
- 不能悄悄改成与旧版本相反的默认策略。

### GetCache

用途：

- 提供语义缓存或近似命中能力。

契约要求：

- 命中与未命中的语义必须稳定。
- 超时、内部错误与未命中必须可区分。
- 具体向量索引、阈值和存储方案可以在服务内部演化，但不应泄漏为新的外部协议依赖。

## 3. 不属于 Python 服务的职责

- HTTP 主链路编排。
- 请求最终路由决策。
- Provider 选择。
- 全局降级策略决策。

这些职责都应留在 Go 侧。

## 4. 兼容性要求

如果要修改 RPC 的字段或语义：

- 先更新 `proto/gateway.proto`。
- 同步更新生成物。
- 运行仓库级 proto 同步检查。
- 更新相关测试与文档。

## 5. 对维护者的提醒

不要把“Python 能做更多事情”误解成“Python 应该主导更多边界”。当前架构选择它做增强能力，是为了让灵活性被控制在稳定接口之后。
